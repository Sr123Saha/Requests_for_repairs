КРАТКАЯ СПЕЦИФИКАЦИЯ ПРОГРАММНОГО МОДУЛЯ
======================================

1. Общие сведения
-----------------

Наименование системы: программный модуль учёта заявок на ремонт климатического оборудования.

Назначение системы: автоматизация приёма, регистрации, обработки и анализа заявок на ремонт климатического оборудования (кондиционеры, вентиляция и др.) в сервисном центре, работающем с заказчиком ООО «Конди».

Цели разработки:
- сокращение времени реакции на обращения клиентов;
- повышение качества обслуживания;
- улучшение планирования загрузки специалистов и контроля статусов заявок.


2. Основные функции
-------------------

Система реализует следующие функции:

- приём и регистрация заявок от заказчиков;
- отображение списка заявок с основными параметрами;
- редактирование параметров заявок (описание, статус, исполнитель и др.) в зависимости от роли пользователя;
- назначение ответственных специалистов;
- ведение истории изменений статуса;
- формирование статистики:
  - количества завершённых заявок;
  - среднего времени выполнения заявки;
  - распределения заявок по типам оборудования;
- управление пользователями (назначение ролей, блокировка/разблокировка);
- генерация QR‑кода со ссылкой на форму оценки качества работы сервиса.


3. Основные сущности и данные
-----------------------------

3.1. Пользователь (users)
- ФИО;
- телефон;
- логин (уникальный);
- пароль;
- роль (тип пользователя);
- признак активности;
- дополнительные контактные данные и заметки.

3.2. Заявка (requests)
- идентификатор и номер заявки;
- дата регистрации;
- тип климатического оборудования;
- модель оборудования;
- описание проблемы;
- статус заявки;
- ссылка на заказчика;
- ссылка на специалиста;
- ссылка на менеджера по качеству (при необходимости);
- даты выполнения и служебные поля (приоритет, история обновлений и др.).

3.3. Дополнительные сущности
- комментарии по заявкам;
- отзывы клиентов;
- комплектующие и их использование в заявках;
- история изменения статусов;
- специализации специалистов;
- уведомления пользователям.


4. Роли пользователей и права доступа
-------------------------------------

Заказчик:
- регистрируется самостоятельно;
- создаёт новые заявки только на себя;
- видит и частично редактирует только свои заявки (дата и описание проблемы).

Специалист:
- видит заявки, к которым он назначен;
- может изменять статус назначенных заявок;
- может добавлять служебную информацию (комментарии, данные по комплектующим).

Оператор:
- создает и редактирует заявки;
- назначает специалистов;
- меняет статусы заявок в рамках своих полномочий.

Менеджер:
- имеет расширенные права по работе с заявками (просмотр и редактирование всех заявок);
- управляет пользователями: может изменять роли и активность учётных записей;
- может создавать новых заказчиков.

Менеджер по качеству:
- обладает правами на просмотр и редактирование всех заявок;
- участвует в продлении сроков выполнения заявок и консультировании по сложным случаям;
- использует отзывы для анализа качества работы.

Администратор:
- техническая роль с полным доступом ко всем сущностям и настройкам (в учебном проекте может не иметь отдельного пользователя, функции администратора фактически выполняет менеджер).


5. Входные и выходные данные
-----------------------------

Входные данные:
- данные форм веб‑интерфейса (логин, пароль, параметры заявок, фильтры);
- параметры URL (идентификатор заявки при открытии карточки или генерации QR‑кода).

Выходные данные:
- таблица заявок с основными полями и статусами;
- подробная карточка заявки;
- статистические показатели (количество завершённых заявок, среднее время, распределение по типам);
- изображение QR‑кода со ссылкой на форму отзыва.


6. Требования к интерфейсу
--------------------------

- единый внешний вид всех форм на основе Bootstrap и кастомных стилей;
- навигационное меню с основными разделами (заявки, новая заявка, статистика, управление пользователями);
- последовательный интерфейс с возможностью возврата к списку заявок;
- на каждой форме указан понятный заголовок, соответствующий выполняемой функции.
